# 项目代号 Echo 路线图

## 产品北极星

把项目代号 Echo 从“有产品方向创新的 AI 写作工具”，升级为“具备作者记忆、环境式智能与持续反馈能力的写作运行时”。

路线图的判断标准不是“又做了多少生成能力”，而是：

- 作者是否更容易继续写下去
- 产品是否越来越懂同一个作者
- 智能是否越来越低打断
- 写作、修改、积累是否被串成同一套流程

## 近期目标：能力放大器成形

阶段目标：把现有写作台做成真正可用的能力增强系统，而不是零散功能集合。

### 重点建设

1. 修改清单与修订工作流
- 把“这里以后再改”正式变成 revision queue
- 支持记录问题、状态、标签与回看

2. 结构诊断器
- 输出短反馈，指出节奏拖慢、转折不足、信息顺序混乱
- 不做整段代写

3. 人物一致性提醒
- 检测人物语气、目标、行为漂移
- 支持长文持续跟踪

4. 写作台整合
- 把灵感、修订、结构、素材入口统一在一个工作台里
- 避免功能散落成“会一点但都不深”的面板集合

### 阶段成果标准

- 用户能在一次写作会话里真正使用修订清单
- 结构与人物反馈不再像零散 AI 结果，而像工作流的一部分
- 写作台不再像功能容器，而像持续陪伴的操作层

## 中期目标：作者记忆引擎落地

阶段目标：把知识库从“可检索资料库”升级成“可持续建模的作者记忆系统”。

### 重点建设

1. 作者记忆图谱
- 自动沉淀人物、关系、场景、母题、意象、时间线
- 建立作品内的长期结构关系

2. 采纳与偏好回流
- 记录用户采纳过的回声和保留过的表达
- 形成个人偏好层

3. 修改记忆
- 识别反复出现的问题模式
- 为后续修订与训练提供基础

4. 素材收纳箱
- 将织带回声、手动摘录、选中文本、外部片段统一沉淀
- 支持轻量标签与后续复用

### 阶段成果标准

- 系统能解释“为什么这个建议更像你”
- 不同功能开始共享同一套作者状态
- 用户感知到产品在长期使用中“越来越懂自己”

## 远期目标：写作运行时成立

阶段目标：从一组功能，升级为一个持续运行的写作系统。

### 重点建设

1. 环境式代理体系
- 织带代理
- 结构代理
- 人物代理
- 修订代理
- 素材代理

2. 文档事件驱动
- 文本更新、停顿、采纳、修订、切章等事件驱动系统状态变化
- 不再以手动触发生成作为默认入口

3. 多界面共享状态
- 织带、详情、写作台、素材池、修订面板共享统一状态源
- 用户看到的是一个系统，不是一堆散功能

4. 成长闭环
- 写作画像
- 微练习入口
- 个人写作能力演进记录

### 阶段成果标准

- 系统从“会响应”升级为“会持续感知”
- 用户把产品当成写作环境的一部分，而不是一个额外工具
- 团队可以清楚地把 Echo 描述为 authoring runtime

## 核心优先级清单

### 第一优先级

- 作者记忆图谱
- 修改清单与修订工作流
- 结构诊断器
- 人物一致性代理
- 织带 2.0 作为环境式智能界面

### 第二优先级

- 素材收纳箱
- 场景卡 / 章节卡
- 仿写陪练
- 写作画像
- 微练习入口

## 明确不优先做什么

- 不优先做重社区
- 不优先做全自动代写
- 不优先做全能聊天框
- 不优先做“模型接得更多”的表层扩展
- 不优先做大量实验性 AI 模块堆叠

这些方向不是永远不做，而是当前会稀释主线。

## 路线图执行原则

- 先做长期状态，再做更多单点功能
- 先做低打断交互，再做显眼 AI 表达
- 先做作者记忆，再做更复杂的智能编排
- 先做工作流闭环，再做功能列表增长

## 对内统一判断

未来每加一个新功能，都要先问四个问题：

- 它是否让作者更容易继续写？
- 它是否让系统更理解同一个作者？
- 它是否能融入现有写作流，而不是打断它？
- 它是否在强化“写作运行时”而不是削弱它？

如果答案是否定的，就不该进入优先队列。
