# RAG 织带检索延迟说明与优化

## 可能耗时环节（按常见占比）

1. **语义扩展（query expand）**  
   当设置中开启「智能扩展查询词」时，会先调用主模型 API，把用户输入扩展成更多相关词（如「春天」→「春天 花开 温暖 春风」），再拿扩展后的文本做检索。  
   - 典型耗时：约 1-2 秒（依赖 API 与网络）。  
   - **优化**：若更在意速度，可在设置里关闭「智能扩展查询词」，织带会直接用原文检索，少一次 API 调用。

2. **查询向量化（embed）**  
   检索前需要把当前查询转成向量，才能和知识库里的向量做相似度计算。  
   - **走嵌入 API**：一般 0.5-2 秒，受网络影响。  
   - **走本地 Transformers.js**：首请求可能 10-30 秒（加载模型），之后单次约 3-15 秒（视设备和模型大小）。  
   - **优化**：  
     - 推荐在设置中配置「嵌入 API」（并填写 Base URL、Key、模型），延迟更稳定。  
     - 已对相同查询做短期缓存（约 60 秒、最多 5 条），重复请求会直接命中缓存，避免重复算向量。

3. **从 IndexedDB 加载 chunks**  
   `getChunksByBase()` 会拉取当前知识库下所有 chunk（含向量）。  
   - chunk 数量多、且带 384/768 维向量时，读取与反序列化会变慢（约 1-5 秒量级）。  
   - **优化**：控制单库内导入文件/分块数量；或后续可考虑「只加载部分 chunk / 按需加载」等策略。

4. **向量相似度 + 关键词打分**  
   在内存里对已加载的 chunk 做向量相似度和关键词打分，通常远快于上面几步（百毫秒级），除非 chunk 数量极大。

## Chunk 策略对延迟的影响

### 分块越大，chunk 数量越少

| 分块大小 | 10万字产生的 chunk 数 | 加载时间 | 检索时间 |
|----------|----------------------|----------|----------|
| 200 字 | ~500 个 | 较快 | 较快 |
| 280 字 | ~360 个 | 中等 | 中等 |
| 500 字 | ~200 个 | 更快 | 更快 |

### 但分块过大也有代价

- **语义稀释**：一个 chunk 包含多个主题，向量变成"平均语义"
- **匹配精度下降**：即使速度快了，搜到的内容可能不相关

### 当前平衡策略

我们采用 **280 字 + 100 字重叠** 的配置：
- 比 200 字更完整，语义更集中
- 比 500 字更精细，匹配更准确
- 100 字重叠（约 50%）保证上下文连贯

### 优化建议

如果你在意速度：
1. **关闭「智能扩展查询词」** — 减少 1-2 秒
2. **配置 API 嵌入** — 避免本地模型加载时间
3. **控制单库文件数量** — 建议一个知识库不超过 20-30 个文件

如果你在意质量：
1. **开启「智能扩展查询词」** — 提升召回率
2. **使用中文友好的嵌入模型** — 如 `BAAI/bge-m3`
3. **配置强制书目** — 确保重要来源必现

## 开发者面板中的耗时日志

在 Dev 面板中可以看到各阶段耗时，便于定位瓶颈：

- `[ribbon] query expand done`：语义扩展耗时（仅开启时出现）。
- `[rag] chunks loaded`：从 IndexedDB 加载 chunk 的耗时与数量。
- `[rag] embed done`：单次查询向量化耗时。
- `[rag] vector score done`：向量检索 + 打分的耗时。

根据哪一项 ms 最大，可针对性选择上述优化方式。

## 重新导入文件后的变化

**注意**：修改 chunk 策略后，需要**重新导入文件**才能生效。

- 旧文件的 chunk 大小不会改变
- 新导入的文件会使用新的分块策略
- 可以在「知识库管理」中删除并重新添加文件
