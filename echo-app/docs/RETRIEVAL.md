# 检索策略说明 (RAG Retrieval Strategy)

## 当前策略概览

织带使用的检索是 **混合检索**：

1. **向量检索（语义）**  
   当知识库中的片段已成功生成 **embedding** 时：
   - 用当前输入文本生成查询向量，与库中每个片段的向量做 **余弦相似度** 计算。
   - 相似度高的片段会优先进入候选。

2. **关键词检索**  
   - 对查询与片段做简单关键词/字符重叠打分（含整句匹配、前两字匹配、单字重叠等），适配中文。

3. **混合与抖动**  
   - 有 embedding 时：`最终得分 = 0.7 × 向量得分 + 0.3 × 关键词得分`。  
   - 无 embedding 时：仅用关键词得分。  
   - 最后对得分施加约 20% 的 **语义抖动**，让结果有一定随机性，避免总是同一批片段。

4. **强制书目**  
   - 若配置了「强制检索」的书籍，会先保证 1～3 条来自这些书，其余槽位从非强制书中按得分选取。

因此：**RAG 本身是支持语义的**，依赖的是「查询向量 ↔ 片段向量」的相似度；关键词是补充，用来抓住字面匹配。

## 为什么「春天」搜不到描写春天但没写「春天」的段落？

可能原因：

1. **没有向量（无 embedding）**  
   - 若 PDF 解析失败、或 embedding 生成失败（如模型加载报错被 catch），片段就没有 `embedding`。  
   - 此时检索会 **退化为纯关键词**，只能匹配到字面含「春天」的句子。

2. **当前 embedding 模型偏英文**  
   - 使用的是 `Xenova/all-MiniLM-L6-v2`，对英文语义更好，对中文「春天」和「花开、春风、三月」等的语义距离不一定近。  
   - 所以即使用向量检索，也可能无法把「描写春天的博尔赫斯段落」和用户输入的「春天」拉近。

3. **查询太短**  
   - 若只拿「春天」两个字去算向量，语义信息有限，相似度排序可能不稳定。

**结论**：  
- 语义能力 = 有 embedding + 模型能理解该语言。  
- 当前在中文上偏弱，所以会显得「笨」、更像关键词匹配。

## 用 AI 增强检索（可选）

可以通过 **AI 查询扩展** 提升语义召回，尤其是中文：

- 用户输入：「春天」  
- 先调用一次 LLM（你配置的 API），生成若干 **同义/相关词或短句**（如：春天、花开、温暖、春风、复苏、三月、博尔赫斯式的花园与时间）。  
- 用「原查询 + 扩展词」一起做 **向量 + 关键词** 检索，这样即使原文没有「春天」二字，但含有「花开」「春风」等，也能被向量或关键词命中。

在设置中开启 **「语义扩展」** 后，每次织带检索前会先做一次轻量级 AI 扩展再查；未配置 API 或未开启时，仍使用原有检索逻辑。

## 嵌入模型配置（API 嵌入）

在 **设置 → 嵌入模型** 中可启用 **API 嵌入**，并配置服务商与模型：

- **硅基流动**：`https://api.siliconflow.cn`，模型如 `BAAI/bge-m3`、`Qwen3-Embedding-0.6B`
- **OpenRouter**：`https://openrouter.ai/api/v1`，模型如 `openai/text-embedding-3-small`
- **AIHubMix**：`https://aihubmix.com`，模型如 `text-embedding-3-small`、`Qwen/Qwen3-Embedding-0.6B`

启用并填写 Base URL、API Key、模型名称后，导入文件与织带检索时的向量均通过该接口生成，中文语义效果更好、更稳定。未配置时仍使用本地 Transformers.js 模型（易失败或效果一般）。

## 嵌入与知识库数据存在哪里？

- **全部在浏览器本地**：
  - **共鸣库元数据**（库列表、文件列表、强制书目等）仍在 **localStorage**：`echo-knowledge-bases`、`echo-active-knowledge-base-id`。
  - **所有片段（chunk）**（含文本与 **embedding 向量**）存放在 **IndexedDB**：数据库名 `echo-knowledge-db`，对象存储 `chunks`（主键 `id`，索引 `baseId`）。
- **容量**：IndexedDB 单源通常可达数百 MB 以上，可支持更多、更大的 PDF 导入与 API 嵌入向量。首次使用时会自动从 localStorage 的旧 key（若存在）迁移到 IndexedDB。

## PDF 显示 0 意象、0 字符

常见原因：

- **扫描版 PDF / 纯图片**：当前没有 OCR，无法提取文字，解析后 chunk 数为 0。  
- 会提示：「该 PDF 可能为扫描件，无法提取文字；请使用含文字层的 PDF」。

若确认是带文字层的 PDF 仍为 0，请查看控制台日志中 `parseResult.chunks.length` 与 `totalChars`，便于进一步排查解析或分块逻辑。
